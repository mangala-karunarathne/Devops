# Kubernetes Sidecar Pattern — Study Material
### Topic: Log Shipping with Shared EmptyDir Volume

---

## 📋 Original Task

> We have a web server container running the nginx image. The access and error logs generated by the web server are not critical enough to be placed on a persistent volume. However, Nautilus developers need access to the last 24 hours of logs so that they can trace issues and bugs. Therefore, we need to ship the access and error logs for the web server to a log-aggregation service. Following the separation of concerns principle, we implement the Sidecar pattern by deploying a second container that ships the error and access logs from nginx. Nginx does one thing, and it does it well—serving web pages. The second container also specializes in its task—shipping logs. Since containers are running on the same Pod, we can use a shared emptyDir volume to read and write logs.

### Task Requirements:
1. Create a pod named `webserver`
2. Create an `emptyDir` volume `shared-logs`
3. Create two containers:
   - `nginx:latest` → container name: `nginx-container`
   - `ubuntu:latest` → container name: `sidecar-container`
4. Add command on sidecar-container:
   ```
   "sh", "-c", "while true; do cat /var/log/nginx/access.log /var/log/nginx/error.log; sleep 30; done"
   ```
5. Mount volume `shared-logs` on both containers at `/var/log/nginx`
6. All containers should be up and running

---

## 🛠️ Solution — Pod Manifest (YAML)

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: webserver
spec:
  volumes:
    - name: shared-logs
      emptyDir: {}

  containers:
    - name: nginx-container
      image: nginx:latest
      volumeMounts:
        - name: shared-logs
          mountPath: /var/log/nginx

    - name: sidecar-container
      image: ubuntu:latest
      command: ["sh", "-c", "while true; do cat /var/log/nginx/access.log /var/log/nginx/error.log; sleep 30; done"]
      volumeMounts:
        - name: shared-logs
          mountPath: /var/log/nginx
```

## 🧠 Concepts Explained

### 1. emptyDir Volume

- A **temporary directory** that is created when a Pod is assigned to a node
- **Shared** between all containers in the same Pod
- **Deleted** when the Pod is removed from the node
- Perfect for logs that don't need long-term persistence

```
nginx-container  ──writes──▶  /var/log/nginx (emptyDir)  ◀──reads──  sidecar-container
```

### 2. The Sidecar Command Breakdown

```sh
sh -c "while true; do cat /var/log/nginx/access.log /var/log/nginx/error.log; sleep 30; done"
```

| Part | Meaning |
|---|---|
| `sh -c` | Launch a shell and execute the following string |
| `while true` | Infinite loop — runs forever |
| `do ... done` | Loop body — everything executed each iteration |
| `cat access.log error.log` | Read and print both log files to stdout |
| `sleep 30` | Wait 30 seconds before next iteration |

> When `cat` prints to **stdout**, Kubernetes captures it automatically — this is why `kubectl logs` shows the nginx log entries.

## 📁 nginx Log Files Explained

### `access.log` — The Visit Register
Records **every HTTP request** nginx receives:

```
192.168.1.1 - - [28/Mar/2026] "GET /index.html HTTP/1.1" 200 615
```

| Field | Meaning |
|---|---|
| `192.168.1.1` | Visitor's IP address |
| `GET /index.html` | What page they requested |
| `HTTP/1.1` | Protocol version |
| `200` | Status code (200 = success) |
| `615` | Bytes transferred |

**Common status codes:**
- `200` → Success
- `301` → Redirect
- `404` → Page not found
- `500` → Server error

### `error.log` — The Internal Issue Diary
Records **nginx internal events** — NOT end user logins:

```
2026/03/28 15:19:49 [notice] start worker process 78
```

- Startup/shutdown notices
- Worker process events
- Configuration issues
- Backend connection failures

### When Does an End User Action Appear in Each Log?

```
End user visits website / logs in
         │
         ├──▶ access.log ✅ ALWAYS — IP, URL, status code
         │
         └──▶ error.log  ❌ ONLY IF something breaks
                              (e.g., DB unreachable, SSL expired)
```

### Log Path Convention

| Path | Scope |
|---|---|
| `/var/log/` | Universal Linux standard for ALL app logs |
| `/var/log/nginx/` | nginx-specific (convention, not hardcoded) |
| `/var/log/apache2/` | Apache web server |
| `/var/log/mysql/` | MySQL database |
| `/var/log/auth.log` | System login/auth events |

## 🏗️ Sidecar Design Pattern

### Is it Kubernetes-Specific?
**No.** It's a general **Microservices Architecture** design pattern. Kubernetes just provides the best infrastructure to implement it.

> Think of it like REST API — it's a design pattern you can implement anywhere. Sidecar is similar.

### The Motorcycle Analogy 🏍️

```
🏍️  +  🛺
Main App    Sidecar
runs its    attached, assists,
own job     doesn't interfere
```

### Container Design Patterns Family

| Pattern | Purpose |
|---|---|
| **Sidecar** | Helper runs alongside main container |
| **Ambassador** | Proxy between container and outside world |
| **Adapter** | Standardizes output of main container |
| **Init Container** | Runs BEFORE main container starts |

### Architecture Diagram

```
┌─────────────────── Pod: webserver ───────────────────┐
│                                                       │
│  ┌─────────────────┐      ┌──────────────────────┐   │
│  │ nginx-container │      │  sidecar-container   │   │
│  │  (nginx:latest) │      │  (ubuntu:latest)     │   │
│  │                 │      │                      │   │
│  │  Serves pages   │      │  Ships logs every    │   │
│  │  Writes logs ───┼──────▶ 30s via cat command  │   │
│  └────────┬────────┘      └──────────┬───────────┘   │
│           │                          │                │
│           └──────────┬───────────────┘                │
│                      │                                │
│             shared-logs (emptyDir)                    │
│              /var/log/nginx                           │
└───────────────────────────────────────────────────────┘
```

## 🚀 Real World — MERN / PERN Application

### How It Maps to Your Stack

```
Pod
├── express-container       → Node.js/Express API
│     writes to /var/log/app/error.log
│     writes to /var/log/app/combined.log
│
├── react-nginx-container   → Serves React build
│     writes to /var/log/nginx/access.log
│     writes to /var/log/nginx/error.log
│
└── sidecar-container       → Ships ALL logs
      reads from shared emptyDir volume
      ships to Elasticsearch / Loki / CloudWatch
```

### What You Can Capture in MERN/PERN

| Failure Point | Log Captured |
|---|---|
| MongoDB connection failed | Express error log |
| JWT token validation failed | Auth error |
| PostgreSQL query timeout | DB error in app log |
| API route crash | Express 500 error |
| React build serving failed | nginx error log |
| Repeated failed logins | access.log pattern |

### Winston Logger Setup (Node.js)

```javascript
const winston = require('winston');

const logger = winston.createLogger({
  transports: [
    new winston.transports.File({
      filename: '/var/log/app/error.log',    // sidecar reads this
      level: 'error'
    }),
    new winston.transports.File({
      filename: '/var/log/app/combined.log'  // sidecar reads this
    })
  ]
});

// Capture failure points
app.get('/api/users', async (req, res) => {
  try {
    const users = await db.query('SELECT * FROM users');
    res.json(users);
  } catch (err) {
    logger.error(`DB Query Failed: ${err.message}`); // captured!
    res.status(500).json({ error: 'Database error' });
  }
});
```

## 📌 Key Takeaways

1. **Sidecar pattern** = separation of concerns — each container does one job well
2. **emptyDir** = lightweight shared storage within a Pod's lifetime
3. **access.log** = user activity; **error.log** = internal nginx events
4. The pattern is **not K8s exclusive** — works in any container orchestration
5. In production, the sidecar ships to real monitoring tools (ELK, Loki, CloudWatch)
6. The same pattern applies directly to **MERN/PERN** apps for capturing failure points
